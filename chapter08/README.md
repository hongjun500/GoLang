# 泛型(Generics)

> `Go` 1.18 引入泛型：在**编译期**用**类型参数**写出既复用、又类型安全的代码，
> 避免过去 `interface{}` + 类型断言/反射带来的样板代码与运行时风险

## 为什么需要泛型

> 泛型出现前，"对不同类型做同一件事"只能三选一，各有代价

| 方案                     | 问题                                 |
| ------------------------ | ------------------------------------ |
| 每种类型复制一份函数     | 重复代码，难维护                     |
| `interface{}` + 类型断言 | 丢失编译期类型检查，运行时可能 panic |
| 反射                     | 慢、复杂、易错                       |

> 泛型同时做到：**一份代码 + 编译期类型安全 + 无反射开销**

## 基本语法

类型参数列表写在方括号里，**函数名/类型名紧贴 `[`（中间无空格）**

```go
// 泛型函数：类型参数在函数名后
func Index[T comparable](s []T, x T) int { ... }
// 泛型类型：类型参数在类型名后
type Stack[T any] struct { items []T }

```

## 约束

> 约束规定了"类型参数能是哪些类型、能做哪些操作"。约束本质是一个描述"类型集合"的接口。

- `any`：等价于 `interface{}`，任意类型（不能做 +、< 等操作）。
- `comparable`：可用 == / != 的类型（能做 `map key`）。
- 联合（union）：`int` | `int64` | `float64`，枚举允许的类型。
- 近似（approximation）~T：匹配所有底层类型为 `T` 的类型。 `~int` 能匹配 `type MyInt int`；而 `int`（不带 ~）只精确匹配 `int` 本身。
- 约束接口：把上面这些组合抽成命名接口复用

```go
type Number interface {
	~int64 | ~float64
}
```

- 标准库现成约束：`cmp.Ordered`（Go 1.21+，可用于 </>）、`golang.org/x/exp/constraints`（`Ordered`、`Integer`、`Float` 等）。优先复用，别自己造轮子

> 约束中的联合成员不能是带方法的接口，也不能含 `comparable` 之外的方法型接口混合

## 类型推断

> 调用泛型函数时，类型实参通常不用手写，编译器自动推断

1. **函数实参类型推断**：从传入实参的类型推断出现在参数中的类型参数。 `SumNumbers(ints)` → 由 `map[string]int64` 推出 `K=string`, `V=int64`
2. **约束类型推断**：当某类型参数只出现在返回值、但其约束有"核心类型"时，可据此补全
   > 只在参数中出现的类型参数才最稳妥。若一个类型参数只出现在返回值且约束是 any，会直接报 cannot infer，必须显式写 F[T1, T2](...)

## 泛型类型与方法

```go
type Stack[T any] struct { items []T }

func (s *Stack[T]) Push(v T)        { s.items = append(s.items, v) }
func (s *Stack[T]) Pop() (T, bool)  { ... }
```

- 使用时需实- 使用时需实例化：`var s Stack[int]` 或 `s := &Stack[string]{}`
- **方法不能声明自己的类型参数**，只能用接收者类型上已有的类型参数

## 底层原理解析

- **实现策略**：GC Shape Stenciling + 字典（dictionary），是介于两种极端之间的"部分单态化"：
  - 不是 C++ 模板那样为每个类型全量生成一份代码（会代码膨胀）；
  - 也不是 Java 那样完全类型擦除（会丢类型信息、需装箱）。
  - Go 按 GC shape（内存布局/指针形状） 分组：布局相同的类型（如所有指针类）共享同一份编译后的代码，运行时再通过一个隐藏的 字典 参数拿到具体类型的元信息（方法、size 等）。
- **性能取舍**：共享代码 + 字典查表会引入间接调用、并限制内联，热点路径下泛型不一定比手写具体类型或 interface 更快。泛型的核心收益是类型安全 + 消除重复，而非性能

## 避坑指南

- **方法不能带类型参数**：func (s S) Do[U any](u U) 非法。需要额外类型参数时改为包级泛型函数。
- **约束接口不能当普通接口用**：`var n Number` 非法；`Number` 只能出现在类型参数约束位置，不能用来声明变量/存值。
- **别把泛型退化成 `interface{}`**：让函数"泛型进、`any`/`interface{}` 出"等于没用泛型（调用方还得断言）。返回值该具体就具体（如索引就返回 `int`）
- `~T` vs `T`：想接受命名类型（`type Celsius float64`）务必用 ~，否则只匹配原始类型
- 约束里的 `+`、`<` 要有依据：类型参数只能做其"类型集合中所有类型都支持"的操作。想用 `<` 就得约束为 `cmp.Ordered`
- `comparable` 的边界：`Go` 1.20 起放宽，接口类型也满足 `comparable`，但若底层动态类型不可比较，运行时 `==` 仍会` panic`。
- 不要为了泛型而泛型（官方建议）：只有当你确实在为多种类型重复实现同一逻辑（尤其是容器、通用算法、工具函数）时才用；能用普通接口清晰表达就别上泛型
